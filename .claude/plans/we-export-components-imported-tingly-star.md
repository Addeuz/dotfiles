# Export compiled Tailwind CSS from `@df/components` into `audit/`

## Context

`@df/components` (`libraries/typescript/components`) recently adopted **Tailwind v4 + shadcn-svelte** (commits `58c6a2bc6`, `f4136c6fb2`, `cb43b4d02d`). The package currently exports the shadcn `Button` and more shadcn components will follow.

Today the package's build (`svelte-package`) produces only `dist/index.js` + `dist/index.d.ts` — **no CSS**. The Tailwind utilities the components rely on are generated by `@tailwindcss/vite` only inside the package's own dev/preview SvelteKit app.

The consumer, `audit/`, is a SvelteKit app that uses Carbon Design + plain CSS with explicit cascade layers. It does **not** have Tailwind installed and we want to keep it that way: introducing Tailwind in audit would mean a postcss/Vite plugin pipeline and content-scanning over a large codebase.

**Outcome:** ship a precompiled, static CSS file from `@df/components` containing the Tailwind utilities used by exported components plus the shadcn theme tokens, and consume it from audit via a single `@import` placed inside a named cascade layer in `audit/src/styles/`.

## Approach

Use the **Tailwind v4 CLI** (not Vite) to compile a dedicated entry CSS in the components package to `dist/styles.css`. The CLI runs from the package root, so its content scanner only sees the components package's own source — utilities used in `audit/` will not (and should not) generate output. The contract is: components ship their own utilities.

The published CSS is intentionally **narrower** than the package's dev-preview `src/routes/layout.css`: we drop the global `@layer base` resets (`* { border-color }`, `body { bg-background text-foreground }`, `html { font-sans }`) because they would conflict with Carbon and the existing audit styles. Theme variables (`:root`/`.dark`) and `@theme inline` stay so utilities resolve correctly. The font import is dropped (audit has its own typography system).

Audit imports the CSS through a thin wrapper file that wraps the import in a named cascade layer (`df-components`), explicitly placed in `layer-order.css` so the cascade is deterministic.

## Critical files

- `libraries/typescript/components/package.json` — add `build:css` script, chain into `package`, add `@tailwindcss/cli` devDep, add `./styles.css` export.
- `libraries/typescript/components/src/styles.css` (NEW) — published Tailwind entry. Differs from `src/routes/layout.css` by dropping `@layer base` and the font import.
- `libraries/typescript/components/src/routes/layout.css` — leave untouched (still drives the dev-preview SvelteKit app).
- `audit/src/styles/components.css` (NEW) — wraps `@import '@df/components/styles.css';` in `@layer df-components`.
- `audit/src/styles/layer-order.css` — insert `df-components` between `carbon-overrides` and `global`.
- `audit/src/routes/+layout.svelte` — add `import '../styles/components.css';` to the existing CSS import block.

## Implementation steps

### 1. Components package — build pipeline

In `libraries/typescript/components/package.json`:

- Add devDep: `"@tailwindcss/cli": "^4.2.2"` (match existing `tailwindcss` major).
- Add scripts:
  - `"build:css": "tailwindcss -i ./src/styles.css -o ./dist/styles.css --minify"`
  - `"dev:css": "tailwindcss -i ./src/styles.css -o ./dist/styles.css --watch"`
- Update `"package"` to: `"svelte-kit sync && svelte-package && pnpm run build:css && publint"` so CSS is written **after** `svelte-package` populates `dist/`.
- Add to `exports`:
  ```jsonc
  "exports": {
    ".": { "types": "./dist/index.d.ts", "svelte": "./dist/index.js" },
    "./styles.css": "./dist/styles.css"
  }
  ```
  `files` already contains `dist`, so the new file ships automatically.

### 2. Components package — published CSS entry

Create `libraries/typescript/components/src/styles.css`:

```css
@import 'tailwindcss';
@import 'tw-animate-css';
@import 'shadcn-svelte/tailwind.css';

@custom-variant dark (&:is(.dark *));

:root {
  /* keep all token declarations from src/routes/layout.css */
}

.dark {
  /* keep all token declarations from src/routes/layout.css */
}

@theme inline {
  /* keep the full @theme inline block from src/routes/layout.css */
}

/* INTENTIONALLY OMITTED:
   - @import "@fontsource-variable/ibm-plex-sans";
   - @layer base { * { @apply border-border outline-ring/50 } body { ... } html { ... } }
   These would override Carbon globally in audit.
*/
```

If a future shadcn component genuinely requires IBM Plex Sans, document that audit consumers must install `@fontsource-variable/ibm-plex-sans` separately — don't smuggle it through the shared CSS.

Confirm the build runs: `pnpm --filter @df/components build` should produce `dist/styles.css` containing utility classes used by `Button.svelte` and the theme tokens.

### 3. Audit — layer ordering

Edit `audit/src/styles/layer-order.css` to add the new layer:

```css
@layer carbon, carbon-overrides, df-components, global, spacing, typography-reset, typography;
```

Layers later in the list win — `df-components` sits above Carbon (so shadcn's tokens beat Carbon defaults on shadcn-rendered DOM) but below `global`/typography (so app-level overrides remain authoritative).

### 4. Audit — wrapper CSS

Create `audit/src/styles/components.css`:

```css
@layer df-components {
  @import '@df/components/styles.css';
}
```

The named layer wraps everything Tailwind produces (including its own internal `base`/`components`/`utilities` sublayers) into a single cascade slot the audit app controls.

### 5. Audit — wire into root layout

In `audit/src/routes/+layout.svelte`, add the import alongside the existing block:

```svelte
import '../styles/carbon.css';
import '../styles/global.css';
import '../styles/layer-order.css';
import '../styles/components.css';
import '../styles/spacing.css';
import '../styles/typography.css';
```

Place `components.css` immediately after `layer-order.css` so the layer name is declared before the layered rules arrive.

## Verification

1. `pnpm --filter @df/components build` — confirm `dist/styles.css` exists and contains:
   - `:root { --background: oklch(...) ... }` token block
   - utility classes used by `Button` (e.g. `.inline-flex`, `.rounded-md`, `.bg-primary`)
   - **no** `body { background-color: var(--background) }` rule (verifies base layer was dropped)
2. `pnpm --filter audit check` — svelte-check passes with the new import.
3. `pnpm --filter audit dev` — open the app:
   - import `Button` from `@df/components` into a test page and confirm it renders styled (rounded corners, primary background).
   - confirm Carbon-styled pages (e.g. an existing case page) look unchanged — `body`, `*` borders, and font are unaffected.
4. DevTools cascade inspection: shadcn classes resolve inside `@layer df-components`; Carbon styles render in `@layer carbon`; both coexist.
5. Build the production audit bundle (`pnpm --filter audit build`) and confirm `dist/styles.css` from the components package is bundled in.

## Pitfalls noted

- **Workspace content scanning**: Tailwind v4 CLI run from the package root scans only that package's source. This is correct — keeps audit free of Tailwind. The trade-off: any class string composed dynamically outside the components package will not generate utilities.
- **Font assets**: dropping `@fontsource-variable/ibm-plex-sans` from the published CSS avoids unresolved woff2 URLs in audit.
- **publint**: a CSS-only export subpath may produce a non-JS export warning; acceptable, this is a documented pattern.
- **Dev iteration**: changes to component `.svelte` source flow through pnpm workspace symlinks for JS, but CSS changes require either re-running `build:css` or running `dev:css --watch` in the components package.
